## 1. フォルダ構成

### 基本原則

- **浅く広く**: フォルダ階層は最大 2〜3 階層まで
- **フォルダはコンテキスト、タグは属性**: 「どこに属するか」はフォルダ、「どんな性質か」はタグで管理
- **完璧を求めない**: 後から整理できることがObsidianの強み

### 推奨パターン：PARA 法

PARA（Projects / Areas / Resources / Archive）は PKM の定番手法：

```
Vault/
├── 00_Daily/          # デイリーノート（日付別自動生成）
├── 01_Projects/       # 期限のある能動的プロジェクト
├── 02_Areas/          # 継続的な責任領域（仕事・健康・学習）
├── 03_Resources/      # 将来参照する知識・資料
├── 04_Archive/        # 完了・非アクティブなもの
├── 05_Inbox/          # 未分類の一時保存
└── 09_Template/       # テンプレート置き場
```

### このVaultでの推奨構成

現在の構成を活かしつつ PARA の考え方を取り込んだ構成：

```
Vault/
├── 00_Daily/          # デイリーノート
├── 01_Projects/       # 進行中プロジェクト（ClaudeCode など）
├── 02_Areas/          # 継続管理領域（Obsidian設定・学習など）
├── 03_Resources/      # 書籍・メモ・参考資料
├── 03_Template/       # テンプレート
└── 99_Archive/        # 完了・アーカイブ
```

### フォルダ設計の判断基準

| 判断 | フォルダ | タグ |
|------|----------|------|
| ノートが1か所にしか属さない | ✅ 向いている | |
| ノートが複数の観点で分類される | | ✅ 向いている |
| 10件以上のノートがまとまる | ✅ 向いている | |
| 横断検索したい属性 | | ✅ 向いている |

## 2. タグ戦略

### 基本原則

1. **小文字統一** – `#プロジェクト` より `#project` に統一するか、日本語なら表記を固定
2. **少なく深く** – 1ノートに 3〜5 タグまで。多すぎると意味が薄れる
3. **階層タグを使う** – `#ツール/obsidian` のようにスラッシュで階層化
4. **フォルダと重複させない** – 同じ情報はどちらか一方で管理

### 推奨タグカテゴリ

```
# ステータス系
#status/inbox      # 未処理・要整理
#status/active     # 作業中
#status/done       # 完了
#status/archive    # アーカイブ済み

# タイプ系
#type/daily        # デイリーノート
#type/memo         # メモ・アイデア
#type/project      # プロジェクト
#type/book         # 書籍メモ
#type/reference    # 参考資料

# 領域系（自分のライフエリアに合わせて）
#area/work         # 仕事
#area/learning     # 学習
#area/health       # 健康・習慣
#area/personal     # プライベート

# ツール系
#tool/obsidian
#tool/claude
#tool/git
```

### タグ vs フォルダ 判定フロー

```
ノートを分類したいとき
    ↓
「このノートは1つのカテゴリにしか属さない？」
    ↓ Yes             ↓ No
  フォルダで管理     タグで管理
                        ↓
              「将来このタグで横断検索する？」
                  ↓ Yes    ↓ No
               タグを使う  書かなくてもOK
```

## 3. デイリーノートのフォーマット

### なぜデイリーノートが重要か

デイリーノートは PKM の「インボックス」。考えを一時置きする場所であり、後から整理する起点になる。

### 推奨テンプレート（詳細版）

```markdown
---
date: {{date:YYYY-MM-DD}}
day: {{date:dddd}}
tags: [type/daily]
---

# {{date:YYYY-MM-DD}} ({{date:ddd}})

## 🎯 今日の最重要タスク（MIT）
> 今日これだけは完了させる3つ

- [ ]
- [ ]
- [ ]

## 📋 タスクリスト
- [ ]  📅 {{date:YYYY-MM-DD}}
- [ ]  📅 {{date:YYYY-MM-DD}}

## 📝 キャプチャ（メモ・気づき）
> 思ったこと・見つけたこと・アイデアを気軽に書く

-

## 🔗 今日関係するノート
- [[]]

## 🌅 振り返り（夕方に記入）
### よかったこと・達成したこと

### 改善できること

### 明日に持ち越すこと
- [ ]
```

### デイリーノートのコツ

- **朝**: MIT（Most Important Tasks）を3つだけ決める
- **日中**: キャプチャ欄にメモを追記していく
- **夜**: 振り返りを5分で記入
- **週次**: 週に1度、デイリーノートを見直して重要メモを専用ノートへ移動

## 4. リンク戦略（知識の網を作る）

### 基本的なリンクの使い方

```markdown
[[ノート名]]               # 基本リンク
[[ノート名|表示テキスト]]  # エイリアス付きリンク
[[フォルダ/ノート名]]      # パス指定（同名ファイルがある場合）
```

### MOC（Map of Content）の作り方

MOC とは「特定テーマのノート一覧」。フォルダの代わりに使えるインデックスノート。

```markdown
# 🗺 学習ロードマップ MOC

## Python
- [[Python基礎]]
- [[Python非同期処理]]

## Obsidian
- [[Obsidianプラグイン一覧]]
- [[Obsidianショートカット]]
```

### リンクのベストプラクティス

1. **書きながらリンクする** – 後でまとめてリンクするより、書きながら `[[` を打つ
2. **バックリンクを活用** – 「このノートはどこから参照されているか」を確認する習慣
3. **孤立したノートを減らす** – 定期的にグラフビューで孤立ノートを見つけてリンク追加

## 5. ファイル命名規則

### 推奨ルール

| タイプ | 命名例 |
|--------|--------|
| デイリーノート | `2026-02-18.md`（日付形式 YYYY-MM-DD） |
| プロジェクト | `ProjectName_概要.md` |
| 書籍 | `書名_著者名.md` |
| メモ | `わかりやすい日本語タイトル.md` |
| テンプレート | `XXX_Template.md` |

### 命名のポイント

- **日本語OK** – Obsidianは日本語ファイル名に対応している
- **スペースは使えるが `_` 推奨** – リンクが壊れにくい
- **日付は先頭に** – ソート順が自然になる
- **避ける文字**: `#`, `[`, `]`, `^`, `|`, `\`, `/`, `:`

## 6. テンプレート設計

### テンプレートで必ず入れる要素

**frontmatter（YAML）**:
```yaml
---
date: {{date:YYYY-MM-DD}}
tags: []
status: inbox
---
```

**タイプ別テンプレート**:

| テンプレート | 主な項目 |
|-------------|---------|
| デイリー | MIT・タスク・キャプチャ・振り返り |
| プロジェクト | 目標・期限・タスク・メモ・完了条件 |
| 書籍 | タイトル・著者・要約・引用・アクション |
| メモ | 内容・関連ノート |
| ミーティング | 参加者・議題・決定事項・ネクストアクション |

## 7. おすすめプラグイン（2025年版）

### コアプラグイン（標準搭載・必ず有効化）

- **テンプレート** – テンプレートの挿入
- **デイリーノート** – 日付別ノートの自動生成
- **バックリンク** – 参照元の表示
- **タグペイン** – タグ一覧
- **アウトライン** – 見出しナビゲーション

### コミュニティプラグイン（推奨）

| プラグイン | 用途 |
|-----------|------|
| **Calendar** | デイリーノートのカレンダー表示（インストール済み） |
| **Tasks** | 全vault横断タスク管理（インストール済み） |
| **QuickAdd** | ワンキーでテンプレート挿入・ノート作成（インストール済み） |
| **Dataview** | ノートをデータベースとして検索・一覧化 |
| **Templater** | テンプレートの動的変数（`tp.date`, `tp.file` 等） |
| **Obsidian Git** | vault全体をGitで自動バックアップ |
| **Linter** | ノートのフォーマットを自動整形 |
| **Outliner** | 箇条書き操作の強化 |

> 現在 Calendar・Tasks・QuickAdd がインストール済み。次のステップは **Dataview** と **Templater** の追加を推奨。

## 8. ワークフロー：キャプチャ → 整理 → 活用

### 理想のサイクル

```
【キャプチャ】デイリーノート or 05_Inbox に素早くメモ
      ↓（週次レビュー）
【処理】メモを適切なフォルダへ移動・リンク追加
      ↓
【整理】MOC を更新・タグを整理
      ↓
【活用】グラフビューで知識の繋がりを発見
```

### 週次レビューのルーティン（15〜30分）

1. デイリーノートを振り返り、重要メモを専用ノートへ昇格
2. 完了プロジェクトを `99_Archive` へ移動
3. MOC・インデックスノートを更新
4. Inbox をゼロにする
5. 来週の重要タスクをデイリーノートに仮記入

## 9. バックアップ戦略

### 推奨方法

1. **iCloud / Dropbox 同期** – 最も簡単。無料枠で対応可
2. **Obsidian Sync** – 公式サービス（有料・$10/月）。デバイス間同期に最適
3. **Git バックアップ** – Obsidian Git プラグインで自動コミット。技術者向け

### 最低限やること

- vault フォルダをクラウドストレージに置く（iCloud/Dropbox）
- 月1回は別ドライブにバックアップ
